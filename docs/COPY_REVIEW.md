# Founder Copy Review

Founder Copy Review is the internal Signal HQ room for version-aware copy
approval.

Route:

```txt
/hq/copy-review
```

The room is protected by the existing Signal HQ password gate. It is not linked
from public navigation and carries noindex metadata.

## Principle

Founder approval applies to one exact copy version.

The stable item id tells us where a piece of copy lives. The content hash tells
us what version was approved. If the text changes, even by one character, the
hash changes and the item returns to founder review.

## Files

```txt
content/hq/copy-review/inventory.json
content/hq/copy-review/founder-state.json
content/hq/copy-review/review-report.md
content/hq/copy-review/founder-guidance.md
```

`inventory.json` is generated by the scanner. `founder-state.json` is the review
ledger. Keep both under version control when the review state matters.

## Commands

```sh
pnpm copy:scan
pnpm copy:review
pnpm copy:report
pnpm copy:guidance
```

`copy:scan` updates the inventory.

`copy:review` scans and exports a report.

`copy:report` writes a Markdown summary of approval coverage and high-risk copy.

`copy:guidance` exports founder comments and principles as a retrieval bank.

## What The Scanner Reads

The v1 scanner reads:

- `src`
- `content/hq`
- `content/atlas`
- `signal-growth`
- text-like launch collateral in `public/brand/collateral`
- the public press kit under `public/brand/press`

It extracts meaningful strings from TS/TSX, Markdown, JSON, HTML, and text
files. It skips generated output, dependencies, lockfiles, and the copy-review
state folder itself.

## Review Flow

1. Run `pnpm copy:scan`.
2. Open `/hq/copy-review`.
3. Start in `Needs review`.
4. Use `A`, `W`, `R`, `D`, `C`, `J`, and `K` for fast review.
5. Add founder comments when a line is rejected, deferred, or marked needs work.
6. Promote important comments into Founder Guidance.
7. Save the best approved examples into Hall of Fame.
8. Commit the changed `content/hq/copy-review` files when the review session is
   meant to become durable.

## Persistence Choice

This uses repo-backed JSON because Signal HQ is already local-first and
file-backed for strategic records. It avoids adding a database just to prove the
governance loop.

The tradeoff is clear: server actions write the JSON ledger in a local or
persistent workspace. A stateless deployment can render the room and accept an
action for the current function instance, but that write is not a permanent
database write. If this becomes a multi-user production workflow, move
`founder-state.json` to the same durable storage family as HQ CRM.

## Limitations

- The scanner is heuristic, not a full semantic copy compiler.
- Stable ids are based on file, line, context, and copy type. Large refactors can
  produce new ids even when the text is unchanged.
- Pre-review notes are rule-assisted brand checks, not an LLM call.
- Cross-repo scanning currently happens only when those repos are pulled into
  this scanner or their inventories are imported.
